# ISO 27001 - Beginner Guide for Pentesters

---

## FIRST Let Me Explain What This Even Is

---

```
Forget everything complicated for a second.

Imagine you run a company.

You have:
  → Customer data
  → Employee records  
  → Financial information
  → Computer systems
  → Servers
  → Emails

Someone needs to make sure all of that is PROTECTED.

But HOW do you know if you are protecting it correctly?

That is exactly what ISO 27001 is for.
```

---

## SO WHAT IS ISO 27001 IN SIMPLE WORDS?

---

```
ISO 27001 is basically a RULEBOOK.

It tells companies:
  "Here is HOW you should protect your information"
  "Here is WHAT security measures you should have"
  "Here is HOW to know if your security is working"

Think of it like a driving test.

  → The driving test has RULES you must follow
  → If you follow them correctly you get a LICENSE
  → The LICENSE proves to everyone you can drive safely

ISO 27001 works the same way:
  → Company follows the security rules
  → An outside examiner comes and checks
  → If they pass → they get a CERTIFICATE
  → The certificate tells customers/partners:
    "This company takes security seriously"
```

---

## WHY SHOULD YOU CARE AS A PENTESTER?

---

```
When you do a pentest for a company that has ISO 27001:

  → They have a LIST of security things they CLAIM to have
  → YOUR JOB is to test if those things actually WORK
  → If they fail → you write it in your report
  → They fix it → their security improves

Simple as that.

Also:
  → Many big companies require ISO 27001
  → If you understand it → clients trust you more
  → You can charge more money
  → You write better reports
  → You look more professional
```

---

## HOW IS ISO 27001 BUILT? (Super Simple Version)

---

```
ISO 27001 has TWO main parts:

PART 1 → The MANAGEMENT RULES
         (How the company organizes their security)

PART 2 → The SECURITY CONTROLS  
         (The actual security measures they must have)

Think of it like a restaurant:

PART 1 = The restaurant MANAGEMENT rules
         → Who is responsible for food safety
         → How do you hire cooks
         → How do you check if food is safe
         → What do you do when someone gets sick

PART 2 = The actual KITCHEN RULES
         → Wash your hands
         → Keep raw meat separate
         → Check temperatures
         → Clean the equipment
```

## PART 1 — THE MANAGEMENT RULES

---

```
There are 7 sections (called CLAUSES):

┌──────────────────────────────────────────────────────────┐
│ CLAUSE 1 → UNDERSTAND YOUR COMPANY                       │
│                                                          │
│ In plain English:                                        │
│ "Know what your company does, what information you have, │
│  who could threaten it, and what you need to protect"    │
│                                                          │
│ Example:                                                 │
│ A hospital knows it has patient records                  │
│ It knows hackers want that data                          │
│ It knows it must protect it                              │
└──────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────┐
│ CLAUSE 2 → MANAGEMENT MUST CARE                          │
│                                                          │
│ In plain English:                                        │
│ "The BOSS must take security seriously, not just the     │
│  IT team. Security must come from the top"               │
│                                                          │
│ Example:                                                 │
│ The CEO must say "Security is important here"            │
│ Not just the IT guy working alone                        │
└──────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────┐
│ CLAUSE 3 → FIND AND PLAN FOR RISKS                       │
│                                                          │
│ In plain English:                                        │
│ "Look for things that could go wrong and make a plan     │
│  to deal with them BEFORE they happen"                   │
│                                                          │
│ Example:                                                 │
│ Risk = Hacker could steal our passwords                  │
│ Plan = Make everyone use strong passwords + 2FA          │
└──────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────┐
│ CLAUSE 4 → HAVE THE RIGHT PEOPLE AND TOOLS               │
│                                                          │
│ In plain English:                                        │
│ "Make sure staff are trained, you have the right tools,  │
│  and everyone knows what they are supposed to do"        │
│                                                          │
│ Example:                                                 │
│ Train employees not to click phishing emails             │
│ Have proper security software                            │
└──────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────┐
│ CLAUSE 5 → ACTUALLY DO IT                                │
│                                                          │
│ In plain English:                                        │
│ "Don't just plan security. Actually implement it"        │
│                                                          │
│ Example:                                                 │
│ Don't just WRITE that you will patch systems             │
│ Actually PATCH them                                      │
└──────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────┐
│ CLAUSE 6 → CHECK IF IT IS WORKING                        │
│                                                          │
│ In plain English:                                        │
│ "Regularly check if your security measures are actually  │
│  doing their job. Don't just assume they work"           │
│                                                          │
│ Example:                                                 │
│ Run tests (like PENTESTS) to see if controls work        │
│ Review logs to see if anything suspicious happened       │
└──────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────┐
│ CLAUSE 7 → FIX PROBLEMS AND GET BETTER                  │
│                                                          │
│ In plain English:                                        │
│ "When you find a problem → fix it → make sure it         │
│  doesn't happen again → always try to improve"           │
│                                                          │
│ Example:                                                 │
│ Pentester finds SQL injection → company fixes it         │
│ Company then adds code review process to prevent it      │
│ happening again in future                                │
└──────────────────────────────────────────────────────────┘
```

## PART 2 — THE SECURITY CONTROLS (Annex A)

---

```
"Annex A" sounds scary. It is just a LIST.

A list of 93 security things a company should have.

Think of it like a HOME SECURITY CHECKLIST:

  * Do you have locks on your doors?
  * Do you have an alarm system?
  * Do you have security cameras?
  * Do you have a safe for valuables?
  * Do your neighbors have a key for emergencies?
  * Do you shred important documents?

ISO 27001 Annex A is exactly that — but for companies.

The 93 controls are split into 4 groups:
```

---

### GROUP 1 — ORGANIZATIONAL CONTROLS

```
What does "Organizational" mean?
→ Company RULES and POLICIES and PROCESSES

Examples in plain English:

  "Do you have a written security policy?"
  → Like a document that says "this is how we do security"

  "Do you know what information you have?"
  → Do you have a list of all your important data/systems?

  "Who is responsible for security?"
  → Is it clear whose job it is to protect what?

  "When you hire outside companies (suppliers) 
   do you check if THEY are secure?"
  → Like checking if the cleaning company 
    that comes at night is trustworthy

  "Do you have a plan if something goes wrong?"
  → What do you do if you get hacked?
```

---

### GROUP 2 — PEOPLE CONTROLS

```
What does "People" mean?
→ Controls about the HUMANS in the company

Examples in plain English:

  "Do you check who you are hiring?"
  → Background checks on new employees

  "Do you train your staff on security?"
  → Do employees know NOT to click suspicious emails?

  "When someone leaves the company..."
  → Do you take back their access?
  → Do you delete their accounts?
  → Do you take back their key card?

  "Do people who work from home do it securely?"
  → VPN, secure wifi, company devices?
```

---

### GROUP 3 — PHYSICAL CONTROLS

```
What does "Physical" mean?
→ Real world, physical security. Not just digital.

Examples in plain English:

  "Can random people walk into your server room?"
  → Is the server room locked?

  "Do you have security cameras?"
  → Are important areas monitored?

  "When you throw away old hard drives..."
  → Do you destroy them properly?
  → Or can someone find them in the trash and read the data?

  "Do employees leave sensitive papers on their desk?"
  → Clear desk policy — lock things away when not in use

  "Can someone just walk in behind an employee?"
  → Tailgating — walking in without a badge
```

---

### GROUP 4 — TECHNOLOGICAL CONTROLS

```
What does "Technological" mean?
→ The actual TECHNICAL security measures

THIS IS WHERE MOST PENTESTING HAPPENS

Examples in plain English:

  "Do your systems have the latest updates/patches?"
  → Unpatched systems = easy to hack

  "Do you use strong passwords and 2FA?"
  → Authentication security

  "Can a normal user access the CEO's files?"
  → Access control — who can see what

  "Is your network split up?"
  → Segmentation — if hacker gets in one area
    can they reach everything else?

  "Do you monitor for attacks?"
  → Logging and monitoring — do you even know
    if you are being attacked?

  "Is your code secure?"
  → Secure coding — is the website/app vulnerable to attacks?

  "Do you use strong encryption?"
  → Is data protected when stored and transmitted?

  "Do you scan for vulnerabilities?"
  → Vulnerability management — do you look for weaknesses?
```

## THE MOST IMPORTANT CONCEPT — RISK

---

```
Everything in ISO 27001 revolves around ONE idea:

                         R I S K

What is risk? Super simple:

  RISK = Something bad that COULD happen

Example risks:
  → Hacker could steal customer data
  → Employee could accidentally delete important files
  → Server could crash and take down the website
  → Someone could break into the office and steal laptops

ISO 27001 asks companies to:

STEP 1: FIND the risks
        → What could go wrong?

STEP 2: UNDERSTAND the risks  
        → How bad would it be if it happened?
        → How likely is it to happen?

STEP 3: DEAL WITH the risks (pick one):
        
        Option A → FIX IT
                   "Put controls in place to reduce the risk"
                   Example: Add MFA to prevent password theft
        
        Option B → TRANSFER IT
                   "Pay someone else to deal with it"
                   Example: Buy cyber insurance
        
        Option C → ACCEPT IT
                   "Acknowledge it exists but decide it's 
                    low enough risk to leave for now"
                   Example: Internal test server low risk
        
        Option D → AVOID IT
                   "Stop doing the risky thing entirely"
                   Example: Stop collecting data you don't need

STEP 4: CHECK if your fixes are working
        → This is where YOU come in as a pentester
        → You TEST if the controls actually work

STEP 5: IMPROVE
        → Fix what doesn't work
        → Keep getting better
```

## WHERE PENTESTING FITS IN

---

```
Now this is the important part for you.

Think of it this way:

Company says:
  "We have a lock on our door (security control)"

Pentester says:
  "Let me try to pick that lock (pentest)"

If you can pick it:
  → The control FAILED
  → Company needs a better lock
  → You write this in your report

If you cannot pick it:
  → The control WORKS
  → You write this too
  → Company has evidence their security is effective

That's literally what you do.

More technical examples:

┌─────────────────────────────────────────────────────┐
│ COMPANY CLAIMS          │ YOU TEST                  │
├─────────────────────────┼───────────────────────────┤
│ We use strong passwords │ Try to brute force login  │
├─────────────────────────┼───────────────────────────┤
│ Our network is split up │ Try to move from one area │
│                         │ to another (pivot)        │
├─────────────────────────┼───────────────────────────┤
│ Staff trained on        │ Send fake phishing email  │
│ phishing                │ see if they click         │
├─────────────────────────┼───────────────────────────┤
│ Our app is secure       │ Test for SQL injection,   │
│                         │ XSS, broken auth etc      │
├─────────────────────────┼───────────────────────────┤
│ We monitor for attacks  │ Do attacks and see if     │
│                         │ they detect you           │
├─────────────────────────┼───────────────────────────┤
│ Physical access is      │ Try to tailgate into the  │
│ controlled              │ office without a badge    │
└─────────────────────────┴───────────────────────────┘
```

## HOW TO WRITE YOUR PENTEST REPORT FOR ISO 27001 CLIENTS

---

```
When you find a vulnerability, instead of just saying:

 BAD WAY:
  "Found SQL injection on login page. Fix it."

 GOOD WAY:
  "Found SQL injection on login page.
   
   This means: An attacker could steal ALL customer data
               from your database without a password.
   
   This breaks: ISO 27001 control for secure coding
                (companies must write secure code)
   
   How to fix: Use parameterized queries in your code.
               Also add a Web Application Firewall.
   
   How serious: CRITICAL — customer data at risk"

The client understands the business impact.
The client knows which rule they are breaking.
The client knows exactly what to fix.

That is a professional report.
```

## ONE IMPORTANT DOCUMENT — THE SoA

---

```
SoA = Statement of Applicability

Long name. Simple idea.

It is just a LIST that says:

"Here are all 93 security controls.
 For each one we say:
   → Do we have this? YES or NO
   → If YES: how did we implement it?
   → If NO: why don't we need it?"

WHY YOU CARE:

Before your pentest → ask the client for this document.

It tells you:
  → Exactly what security they CLAIM to have
  → Your job = go and TEST those claims
  → If they claim to have network segmentation
    → go test if you can actually break through it

It is basically your PENTEST SHOPPING LIST.
```

## THE CERTIFICATION — HOW IT WORKS

---

```
Companies go through this process:

STEP 1: GET READY
        Company sets up all their security
        Follows all the rules
        Creates all the documents

STEP 2: INTERNAL CHECK (like a practice test)
        Company checks themselves first
        Fix any problems they find

STEP 3: STAGE 1 AUDIT (document check)
        An outside examiner comes in
        Just reads through all the documents
        Checks everything looks good on paper
        Says "okay you are ready for the real test"

STEP 4: STAGE 2 AUDIT (real test)
        Examiner comes back
        Checks that security is ACTUALLY implemented
        Not just written down
        Interviews staff
        Looks at evidence
        THIS IS WHERE PENTEST REPORTS ARE USED AS EVIDENCE

STEP 5: GET CERTIFICATE
        Pass the audit → get the certificate
        Certificate lasts 3 years

STEP 6: ANNUAL CHECK-UPS
        Every year the examiner comes back
        Makes sure they are keeping up with security
        
STEP 7: RENEW AFTER 3 YEARS
        Full audit again
        Start the cycle over
```

## KEY WORDS YOU WILL HEAR — EXPLAINED SIMPLY

---

```
ISMS
→ Information Security Management System
→ Just means: "The whole system a company uses to manage security"
→ Like saying "the company's security program"

ANNEX A
→ The list of 93 security controls
→ Like a security checklist

CONTROL
→ A security measure
→ Example: "A firewall is a control"
→ "MFA is a control"
→ "Security training is a control"

RISK
→ Something bad that could happen
→ "Risk of data breach"
→ "Risk of unauthorized access"

VULNERABILITY
→ A weakness that could be exploited
→ "Unpatched software is a vulnerability"
→ "Weak password is a vulnerability"

THREAT
→ Something or someone that could exploit a vulnerability
→ "Hacker is a threat"
→ "Phishing email is a threat"

ASSET
→ Anything valuable that needs protection
→ "Customer database is an asset"
→ "Server is an asset"
→ "Employee knowledge is an asset"

AUDIT
→ A formal check to see if you are following the rules
→ Like a school inspection

CERTIFICATION BODY
→ The company that does the official audit
→ Examples: BSI, SGS, Bureau Veritas
→ They are the "examiners"

CLAUSE
→ Just a section/chapter of the standard
→ Clause 4, Clause 5 etc.
→ Think of them as chapters in a book

SoA
→ Statement of Applicability
→ A document listing all 93 controls
→ Says YES or NO to each one with explanation

NONCONFORMITY
→ When something does NOT meet the requirements
→ Like failing a test
→ "We found a nonconformity — your passwords are too weak"

CORRECTIVE ACTION
→ What you do to fix a nonconformity
→ "We fixed it by enforcing strong password policy"
```

## WHAT TO LEARN NEXT

---

```
Now that you understand ISO 27001 basics:

STEP 1: Learn basic networking first
        → How does TCP/IP work?
        → What are ports and protocols?

STEP 2: Learn basic hacking concepts
        → What is a vulnerability?
        → What is exploitation?
        → Learn with TryHackMe or HackTheBox

STEP 3: Come back to ISO 27001
        → Look at the 93 controls again
        → Now they will make more sense
        → You will understand WHAT you are testing

STEP 4: Practice writing reports
        → Every finding needs:
          - What you found
          - Why it matters (business risk)
          - How to fix it

STEP 5: Learn related topics
        → OWASP Top 10 (web vulnerabilities)
        → NIST framework (another security framework)
        → CVE (how vulnerabilities are numbered/tracked)

Remember:
  Good pentester = Technical skills + Reporting skills
  ISO 27001 knowledge makes your reporting much better
  That is what separates you from average pentesters
```
